# 🔍 3rd-review

### 一套 AI agent 糊弄不过去的「异源独立代码审查」

*Claude 审 Claude，几乎必过。那种「过」一文不值。这个工具就是来治它的。*

[English](./README.md) · [简体中文](./README.zh-CN.md)

</div>

---

## 一句话讲清痛点

AI agent 审自己写的代码，会继承自己的盲区——所以基本都说「没问题」。同一个模型既当运动员又当裁判，那个绿色的「通过」是**表演，不是质量门禁。**

`3rd-review` 从结构上保证审查独立：让**另一个**引擎（比如 `codex`、`gemini`）出裁决，实在没有外部引擎，至少也派一个全新、上下文隔离的子代理来审。**写代码的人永远不给自己的卷子打分。**

> 我们日志里的真实案例：一个作者**自己已经审成 `pass`** 的改动，换异源审查员真审，**跑了 6 轮**才真过——finding 数 `5 → 2 → 1 → 1 → 1 → 0`，扒出了一个失败时反而放行的 CI、一条绕过门禁的直推路径、还有一个根本对不上的 schema。**自审给的 `verdict=pass`，等于啥都没说。**

## 你为什么需要它

| 你的情况 | 没有 3rd-review | 有了 3rd-review |
|---|---|---|
| agent 自己写自己审 | 自己放行，盲区照样在 | 裁决来自**独立**引擎 |
| 大体量机械改动 | 一个审查员全读完，烧 token | 机械部分拆给**子审查员**并行 |
| 小文档改两行 | 还跑全套第三方审查，过度 | 自动降级到便宜的*同源隔离*审查 |
| 审查员没完没了挑刺 | 死循环修不完 | **识别靶心漂移**，升级给人工 |
| agent 手写假的 review JSON | 门禁直接放行 | **留证硬化**的执行证明能抓到 |

## 核心思路：审查是「成本梯度」，不是一把锤子砸到底

最重要的设计认知——而且是踩坑踩出来的：**子代理不是「更强」的审查，是「更便宜」的审查。** 独立性是地板，成本才是那个旋钮。

所以每次审查都按内容类型 + 改动量 + 风险，自动路由到三档之一：

```
        越独立 / 越贵
  ▲
  │  R1  异源 + 子审查员       ← 大 diff、大量机械阅读
  │      （外部引擎主审，子代理并行读大头）
  │
  │  R2  异源、单审查员        ← 中等代码 / 方案 / 设计
  │      （单个外部引擎，比如 codex）
  │
  │  R6  同源干净子代理        ← 纯文档、极小改动、或没有外部 CLI
  ▼      （全新隔离上下文——仍然独立，只是同模型）

        越便宜 / 越轻
```

**硬地板——无论如何都不降级：** 命中 `auth` / `migration` / `delete` 关键词 + 真实代码 diff + 改了关键流程规则文件的，首轮**强制**走最重档。风险只会把档位**往上推**，绝不往下。

路由逻辑是一个**纯函数**（`scripts/route-review.mjs`），读一张**纯数据表**（`config/route-rules.json`）——同入同出，没有藏着的宿主状态。薄壳 SKILL.md 从不复述阈值，因为「复述代码的 prose」必然和代码漂移失同步。

## 两种跑法

```
                    ┌─────────────────────────────┐
   绑定了活跃 task？  │  gated adapter（平台内）     │   裁决落盘进 task 目录，
            是 ─────►  review-dispatch-adapter.sh  │   受下游 gate 校验
                    └─────────────────────────────┘
        │ 否
        ▼
   ┌──────────────────────┐
   │  standalone.sh        │   干净环境：审任意代码/文档，
   │  （脱平台）           │   无 gate、无 journal，走退出码契约
   └──────────────────────┘
```

大多数人想要的是 **standalone** 入口——指向一个文件或一个 diff，拿回一个独立裁决。

> **前置条件（第一次跑之前先看）：** `standalone.sh` 需要一个*审查 runner*——也就是真正驱动审查引擎的那条命令。在 agenthub 平台内，这个 runner 由 `review-dispatch-adapter.sh` 自动接好。但在普通的 GitHub checkout 里**不带**那个 adapter，所以你**必须**用 `--review-runner` 传自己的 runner，否则一跑就当场升级人工（escalate）。

```bash
# 独立 checkout：传你自己的审查 runner
# （任何能读 prompt、返回 verdict JSON 的命令——比如一个 codex/gemini 包装脚本）。
./standalone.sh \
  --input=my-change.diff \
  --output-root=./reviews \
  --review-runner='你的审查 runner 命令'

# 退出码契约：
#   0 = 通过(pass)          2 = 升级人工(escalate_to_human)
#   1 = 需返修(revise)      其它 = 执行错误
```

两个入口共享**同一套**审查策略和**同一套**判定脚本。只有环境差异（有没有 gate、有没有 journal）收进两个薄适配器里。注意：`review-dispatch-adapter.sh` 是 agenthub **平台内**用的 adapter，**不在**这个独立仓里。

## 凭什么相信这个裁决

`pass` 不是终点，也不是张嘴就能喊的。三件事保证它诚实：

**1. 独立性不容谈判。** 最终裁决**必须**来自隔离上下文。主 agent 给自己的活打分，正是这个工具存在的全部理由——就是来防它的。

**2. `pass` 必须带证据。** 每个 `pass` 都得带三个字段：
- `reviewSnapshot[]`——每个被审文件的 `path / gitHead / mtime / hash`（客观，承载覆盖率证明；持久化层*可以*从审查员的 read-set 回填）。
- `worktreeInventory`——`included / unrelated / excluded` 路径清单（客观，可从 `git status` 回填）。
- `riskDisposition[]`——每个高风险项的 `risk / checkedSource / decision / whyNotBlocking`。这一个是**主观、绝不回填**的——审查员**必须**自己产出，缺了直接 fail-fast。替主观判断回填等于伪造，所以流水线拒绝补。

**3. 留证硬化的执行证明。** 每次真实审查都往一个只追加的账本里盖一个 `_execNonce` + `reviewRecordHash`。门禁交叉校验它俩，已消耗的 nonce 不能重放。*（对自己的边界很诚实：单进程共享 shell 下这是**留证硬化、不是防伪**——真正恶意、有磁盘权限的 agent 能伪造账本。真防伪要靠进程隔离。我们不假装能做到。）*

## 任何路由档位都绕不过的硬护栏

1. **最低回归覆盖**——每轮覆盖本 phase 全部改动行的 ≥80%。
2. **高风险维度必须全审**——不抽查、不降级。
3. **缩范围不达标就回退全量**——缩范围审查任一护栏不满足，**立即**回退全量。
4. **独立性保证**——最终裁决必须来自独立上下文，绝不自审。

## 没人提前警告你的「升级」难题

我们没给审查轮次设固定上限——而且是踩了坑才明白**为什么不能设**。

天真的熔断器挡不住**靶心漂移**：当 agent 每轮换一个领域报 blocking——编译错 → 路径 → scope → 溯源 → 契约 → schema——那个「同一 finding 重复 N 次才熔断」的判定**永远不触发**。挤牙膏式的返修循环，从设计缝隙里溜走了。

> 实测：一个 checkpoint 跑了 **13 轮、0 通过、烧掉约 80 分钟。** 还有一条过于激进的降级规则——便宜档一抓到 ≥1 个 blocking 就立刻升回最贵档——导致严格的 **R6 ↔ R1 锯齿振荡，46% 的轮次仍在跑最贵模式。**

所以升级是**识别漂移**的：它在「同一个**未解决** finding **累积到阈值**」时才触发，能认出「标准每轮上移」的挤牙膏，按 **finding 数 + 严重程度**降级——而不是按固定次数或关键词黑白名单。*总成本 = 单轮成本 × 轮数；轮数失控比单轮贵更烧钱，所以先治循环，再优化单轮。*

## 我们一开始踩错的坑（省得你再踩）

这个工具是一堆淤青结出来的痂。几个塑造了设计的：

- **「脱平台审查总是悄悄选便宜的同源档。」** 根因有三层：薄壳把*「走哪个入口」*和*「用哪种审查方式」*搞混了（这俩是**正交**的）；agent 喂的是*文字描述*而不是真实 diff（被正确判成极小文本记录 → 便宜档）；环境探测从来没真去查 `codex` 装没装。**教训写进设计：先 `command -v codex` 探测，永远喂*真实 diff*——别用「请审查我的 XYZ 方案」这种自然语言。**

- **审查里贵的不是「发现问题」，是「重读」。** 我们扒了一轮真实的 `codex` pass：**343 秒、125 万 token、44 条命令。** 24 条读文件里，**9 条在重读 task 期内绝不变的固定协议文件**（CLAUDE.md、各种契约、schema——每个 220–260 行，每轮从头读）。一个零 finding 的 `pass` 轮，花的时间和 `revise` 轮一样多，因为协议重读跟「有没有问题要找」根本脱钩。**教训：判 ROI 必须扒真实会话，不能凭印象。**

- **「越强 = 越多子代理」的直觉。** 错。把所有上下文都甩给一个外部审查员，子代理一点 token 都省不下来。只有每个子代理读*自己分到的*文件、返回摘要，省 token 才真正落地。

## 仓库结构

```
SKILL.md                  # 主 agent 读的薄壳（只有骨架）
standalone.sh             # 脱平台入口（你大概率要的就是它）
scripts/
  route-review.mjs        # 纯函数路由器——大脑
  verdict-core-hash.mjs   # 留证硬化的哈希
  ...                     # 成本对比、replay diff、报告渲染……
config/
  route-rules.json        # 阈值的唯一权威源
references/               # 详细规则，由子代理/脚本按需加载
golden/                   # 路由器的 golden 输入→预期 fixture
__fixtures__/             # finding 分类 + pass 覆盖率 fixture
```

架构是刻意的：**主 agent 只读薄壳。** 所有详细规则都放在 `references/`，由真正需要的子代理和脚本按需拉取——这样编排上下文永远不会膨胀。

## 设计哲学，四句话

- **独立性是地板，成本是旋钮。** 便宜的审查可以，自审不行。
- **风险只往上路由。** 拿不准的时候，审得更狠。
- **代码管阈值，prose 管意图。** 复述代码的 prose 会和代码漂移。
- **`pass` 是一个必须带证据的主张**——快照、风险处置、清单，外加一个不能重放的 nonce。
